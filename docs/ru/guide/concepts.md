# Ключевые понятия

Понимание этих концепций сделает остальную документацию гораздо яснее.

## Жизненный цикл устройства

Каждое устройство в MYXON проходит определённый жизненный цикл:

```
  pre_registered ──► online / offline ──► claimed ──► (в работе)
       │                                     ▲
       └── Дилер регистрирует SN             │
                                        Клиент забирает
```

| Состояние | Значение |
|-----------|----------|
| `pre_registered` | Серийный номер есть в БД; агент ни разу не подключался |
| `online` | Агент подключён, туннель активен |
| `offline` | Агент был онлайн, но не слал heartbeat >30 секунд |
| `claimed` | Тенант-клиент взял устройство во владение (через QR или код активации) |

## Состояние владения vs состояние связи

Это два независимых измерения:

- **Состояние владения** — кто владеет устройством? (`ready_for_transfer` / `claimed`)
- **Состояние связи** — устройство сейчас онлайн? (`online` / `offline`)

Устройство может быть **claimed, но offline** (клиент владеет, но питание выключено) или **online, но неактивированным** (агент работает, но клиент ещё не активировал).

## Архитектура туннеля

MYXON использует [FRP (Fast Reverse Proxy)](https://github.com/fatedier/frp):

- **frps** работает на облачном сервере MYXON, слушает порт 7000
- **frpc** работает на Orange Pi рядом с агентом
- Каждое устройство получает уникальный **туннель-порт** (напр. 10001) из пула
- Когда вы открываете сессию Web-доступа, backend проксирует ваш браузер к `frps-host:tunnel-port`

Это значит, что на стороне фермы **не нужен ни VPN, ни статический IP**. Orange Pi нужен только исходящий доступ в интернет.

## Публикуемые ресурсы

Устройство может выставлять несколько **ресурсов** — сервисов, которые агент обнаружил в LAN:

```json
[
  { "id": "remote-plus", "protocol": "tcp",  "host": "192.168.1.100", "port": 5843, "name": "HOTRACO Remote+" },
  { "id": "vnc",         "protocol": "vnc",  "host": "192.168.1.100", "port": 5900, "name": "VNC" },
  { "id": "http",        "protocol": "http", "host": "192.168.1.100", "port": 80,   "name": "Web UI" }
]
```

Каждый ресурс получает свой сдвиг туннель-порта (`assigned_port + 0`, `+1`, `+2`, …).

## Политики доступа

Политики доступа позволяют администраторам-клиентам ограничивать, что могут конкретные пользователи. Политики действуют по площадке и по пользователю:

| Разрешение | Чем управляет |
|------------|---------------|
| `allow_hmi` | Открывать HMI-сессии (HOTRACO Direct HMI) |
| `allow_vnc` | Открывать VNC-подключения |
| `allow_http` | Открывать HTTP/веб-сессии |
| `allow_alarms_view` | Видеть панель аварий |
| `allow_alarms_acknowledge` | Подтверждать активные аварии |
| `allow_audit_view` | Видеть журнал событий |

По умолчанию (политика не назначена): все разрешения выданы.

## Модель тенантов

Каждый пользователь принадлежит **тенанту** — изолированной организации. Дилеры и клиенты — в разных тенантах:

```
  Тенант дилера (напр. «AgriTech BV»)
    ├── Пользователи: сотрудники дилера
    └── Устройства: зарегистрированные этим дилером

  Тенант клиента (напр. «Ферма Норд»)
    ├── Пользователи: фермер + техники
    ├── Площадки: «Коровник A», «Коровник B», «Теплица 1»
    └── Устройства: забранные из парка дилера
```

Поле `dealer_id` на записи устройства **никогда не очищается** — даже после того, как клиент его забрал. Так дилер сохраняет видимость парка, не имея доступа к данным клиента.
